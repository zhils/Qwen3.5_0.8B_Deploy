## 分析

### 内存管理
1. **按需分配**: KV cache 随 seq_len 增长按需分配页面
2. **线性增长**: 每 64 tokens 分配 1 个页面，每个页面 6 MB
3. **最大容量**: 128 页面 × 6 MB = 768 MB 总容量

### 页面利用率
1. **seq_len=2048**: 使用 32 页面（25% 容量）
2. **页面大小**: 64 tokens/页面，平衡了粒度和开销
3. **内存效率**: 相比预分配方式，节省 75% 内存

### 回收机制
1. **立即释放**: release_sequence 后立即释放页面
2. **GPU内存观测不变**: `cudaMemGetInfo` 视角下 GPU used 仍为 1906 MB，说明显存可能被 CUDA allocator / memory pool 保留而非真正返还给驱动
3. **重用能力**: 释放的页面可立即被新序列使用


## 结论

Paged KV 实现了高效的内存管理：
1. **内存效率**: 相比预分配节省 75% 内存
2. **灵活分配**: 按需分配，支持动态增长
3. **快速回收**: 页面可立即重用

需要特别说明：
1. **逻辑回收已生效**: `kv_bytes_mb` 从 192 MB 回落到 0 MB，说明分页管理层的资源计数与释放逻辑正确
2. **驱动视角显存未下降**: `gpu_used_mb` 没有同步下降，不代表页面不可复用，而是当前 benchmark 观测到了 CUDA 内存池保留行为
3. **后续优化方向**: 若要体现“物理显存回收”，需要进一步引入更细粒度 allocator 策略或切换显存观测口径
